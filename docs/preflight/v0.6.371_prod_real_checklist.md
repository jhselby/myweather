# v0.6.371 — real per-row Prod trajectory for mae_over_time chart

**Target ship:** Tue 2026-07-21 AM (weekly limit reset day)
**Type:** analysis + frontend (no collector deploy needed — mae_over_time runs in the digest)
**Fixes:** honesty gap softened in v0.6.370a layer-summary block. Restores the invariant "Production = what users saw, aggregated per row via applied_layer stamp" across BOTH lenses.
**Parallels:** v0.6.369 chp/clp attribution wiring for the per-band tables. Same architectural pattern applied to the over-time chart.

---

## Why this ship

`mae_over_time.py:62` currently defines `STRICT_LAYER_KEYS = [..., ("prod", "error_l4")]` — meaning `series[field]["prod"]` in the JSON is literally L4's daily MAE. That's Fitter-era code from before specialists existed ("final applied = L4"). For fields where a specialist supersedes L4 (sr → Lsr; cc/cm → Lc; ch → chp; cl → clp) the frontend elides the misleading "prod" line and marks the specialist line as `isProd:true` — but that specialist line only averages rows where the gate FIRED, so its n differs from raw/L2/L3 on the same chart. Not comparable.

The ch persistence 14-day watch (day 2 today) reads this chart to detect drift, so this matters right now.

**Fix:** add a `prod_real` series to `mae_over_time.json` that reads `error_{applied_layer}` per row (parallel to `decay_fit.py:712-729` for per-band tables). Frontend swaps `prod_real` in as `isProd:true`; specialists demote to intermediate lines. Backward-compat: keep the old "prod" key populated so any external reader doesn't break.

## Site 1 — `analysis/mae_over_time.py` backend

### 1a. Add prod_real accumulator

Current buckets structure (line 107):
```python
buckets = defaultdict(lambda: {ln: [] for ln, _ in LAYER_KEYS})
```
Add a parallel bucket for prod_real:
```python
prod_real_buckets = defaultdict(list)   # (field, day) -> [error_applied per row]
```

In the pair-log iteration loop (starts line 108), after the existing per-layer accumulation but before the `continue`, add:
```python
# Real per-row prod aggregation — reads applied_layer stamp + error_{applied}
# for each pair row. Parallel to decay_fit.py:712-729. Only rows with an
# explicit applied_layer stamp contribute (pre-v0.6.269 rows are stampless
# and would need the fallback walk from decay_fit.py:715-721; skipping them
# is fine because those rows aged out of the 30-day pair log by 07-31).
applied = r.get("applied_layer")
if applied:
    e_applied = r.get(f"error_{applied}")
    if e_applied is not None:
        # For wd (circular), error_applied is already circular-angular via
        # forecast_error_log.py:212. For all other fields, it's linear.
        # Bucket in absolute value (MAE-shaped).
        prod_real_buckets[(fld, day)].append(abs(float(e_applied)))
```

### 1b. Emit prod_real in fresh_series

After the existing series aggregation loop (near line 155-ish, where fresh_series is built from buckets), add:
```python
# Reduce prod_real buckets to per-day n + MAE
for (fld, day), errs in prod_real_buckets.items():
    if len(errs) < MIN_N_PER_DAY:
        continue
    fresh_series.setdefault(fld, {}).setdefault("prod_real", {})[day] = {
        "n": len(errs),
        "mae": round(sum(errs) / len(errs), 4),
    }
```

### 1c. L1_ONLY_FIELDS handling

`L1_ONLY_FIELDS = {"wd"}` at line 51. Pre-wdp, wd rows have no `applied_layer` stamp (excluded per `forecast_snapshot.py:207-212` — see `scratchpad/wdp_preflight_checklist.md` site 3). So the `if applied:` guard above naturally no-ops for wd — no explicit branch needed. Post-wdp (07-27+), wd rows will start carrying `applied_layer` and prod_real will populate for wd automatically.

**Decision:** no wd-specific branch. The `if applied:` gate is the whole story.

### 1d. Keep old "prod" key for backward-compat

Do NOT delete `("prod", "error_l4")` from STRICT_LAYER_KEYS. Any external consumer of `mae_over_time.json` (nothing I know of, but be safe) continues to see the old series. Frontend just stops reading it. Deprecation cleanup can be v0.6.372 or later.

## Rollback path

If prod_real produces wonky numbers (e.g. missing applied_layer stamps on a chunk of rows, or a bug in the accumulator):
1. Revert `FIELD_LAYERS` changes in corrections_debug.html to point isProd back at the specialist/prod layers.
2. Comment out the prod_real accumulator + emission in mae_over_time.py.
3. Rerun the digest — prod_real key stops populating; frontend already ignoring it.
4. Existing "prod" series continues to work (was never removed).
